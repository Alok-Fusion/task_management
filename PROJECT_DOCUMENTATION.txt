1. PROBLEM STATEMENT
================================================================================

Modern software teams — especially distributed or hybrid ones — struggle with
several interconnected challenges:

 a) TASK VISIBILITY CHAOS
    Team members have no single source of truth for what is being worked on,
    what's overdue, and who is responsible for what.

 b) COMMUNICATION OVERHEAD
    Important events (new task assignments, project updates, member joins) are
    communicated via external chat tools, which leads to missed notifications
    and context switching.

 c) TEAM BURNOUT BLINDNESS
    Managers have no way to proactively detect when a team member is
    overloaded with tasks until it becomes a performance issue. Burnout is
    reactive, not preventive.

 d) AUDIT TRAIL GAPS
    There is no history of who did what and when in a workspace. This makes
    debugging project failures and reviewing team activity impossible.

 e) TASK CREATION FRICTION
    Writing good task descriptions takes time. Team members often create
    vague tasks like "Fix the bug" without any actionable detail, leading to
    confusion and re-work.

 f) PERMISSION INFLEXIBILITY
    Most task managers have rigid permission systems. A task assignee who
    wants to simply mark their task as "done" often cannot do so without
    full edit permissions — creating unnecessary friction.


================================================================================
2. SOLUTION OVERVIEW
================================================================================

Team Sync is a B2B collaborative workspace and task management platform that
solves the above problems with 6 key innovations:

 ✅ UNIFIED WORKSPACE
    A single platform for all team activity — projects, tasks, members,
    and analytics — organized in shareable workspaces.

 ✅ PROACTIVE EMAIL NOTIFICATIONS
    Every meaningful event (task assigned, project created, member joined,
    role changed) automatically triggers a branded email to all relevant
    parties, removing the need for manual communication.

 ✅ TEAM BURNOUT RADAR
    A real-time health scoring system that computes a wellness score (0-100)
    for each team member based on their overdue tasks, high-priority load,
    and workload relative to team average. Overloaded members receive
    automatic email alerts.

 ✅ LIVE ACTIVITY FEED
    A paginated, real-time audit timeline of every action in the workspace —
    who created what, when, with what status. Auto-refreshes every 10 seconds.

 ✅ AI-POWERED TASK ENHANCEMENT
    Uses Google Gemini 1.5 Flash to turn a simple task title into a
    professional description with priority suggestion and subtask checklist,
    directly in the task creation form.

 ✅ SMART PERMISSION SYSTEM
    Role-based access control with a special bypass: task assignees can always
    update their own task's status, even without the EDIT_TASK permission.

LAYER RESPONSIBILITIES
──────────────────────

  CLIENT LAYER
  • Renders UI components (shadcn/ui + Tailwind CSS)
  • Manages server state with TanStack Query (caching, refetching, mutations)
  • Polls the server every 5–30 seconds depending on the data type
  • All API calls go through Axios client at /src/lib/axios-client.ts
  • Authentication state stored in React context

  SERVER LAYER
  • Express.js handles all HTTP routes
  • Passport.js manages authentication strategies (Local + Google OAuth)
  • cookie-session stores session tokens server-side
  • Zod validates all incoming request bodies
  • asyncHandler wraps controllers, passing errors to Express error handler
  • roleGuard checks user's workspace role against required permissions

  DATA LAYER
  • MongoDB Atlas (cloud-hosted)
  • Mongoose ODM for schema definitions and query building
  • TTL index on Activity collection (90-day auto-expiry)

  EXTERNAL SERVICES
  • Gmail SMTP via Nodemailer for transactional emails
  • Google OAuth 2.0 for social login
  • Google Gemini 1.5 Flash for AI task enhancement

  User fills Create Task Form
          │
          ▼
  [Optional] Click "✨ AI Enhance"
          │
          ├──► POST /api/ai/enhance-task
          │         │
          │         ▼
          │    Google Gemini API
          │         │
          │         ▼
          │    Return: description, priority, subtasks
          │         │
          │         ▼
          │    Form fields auto-filled
          │
          ▼
  Submit form → POST /api/task/project/:pid/workspace/:wid/create
          │
          ▼
  Zod schema validation
          │
          ▼
  roleGuard → check CREATE_TASK permission
          │
          ▼
  createTaskService → Mongoose save to MongoDB
          │
          ├──► logActivity("task_created", ...)  →  MongoDB Activity
          │
          ├──► (if assignedTo) sendEmail(assignee, taskAssignedTemplate)
          │         └──► Gmail SMTP (async, fire-and-forget)
          │
          ▼
  Response: { task }
          │
          ▼
  TanStack Query invalidates ["all-tasks", workspaceId]
          │
          ▼
  Task table re-fetches and updates UI


─── BURNOUT RADAR FLOW ───────────────────────────────────────────────────────

  Dashboard loads "Team Burnout Radar" tab
          │
          ▼
  GET /api/analytics/workspace/:wid/health
          │
          ▼
  Fetch all members in workspace
          │
          ▼
  MongoDB Aggregation on Tasks:
          ┌── for each member:
          │   ├── totalOpen = count where status ≠ DONE
          │   ├── overdue   = count where status ≠ DONE AND dueDate < now
          │   └── highPriority = count where priority = HIGH AND status ≠ DONE
          └──────────────────────
          │
          ▼
  For each member, compute healthScore:
    score = 100
    score -= overdue × 15
    score -= highPriority × 8
    score -= load-ratio penalty (0 / 10 / 20)
    score = clamp(0, 100)
          │
          ▼
  If score < 50 AND last alert > 24h ago:
    sendEmail(member, burnoutAlertTemplate)   →  Gmail SMTP
    store timestamp in memory map
          │
          ▼
  Return sorted health array (lowest score first)
          │
          ▼
  Client renders SVG rings + status badges
  Auto-refetches every 30 seconds


─── EMAIL NOTIFICATION FLOW ──────────────────────────────────────────────────

  Any controller action (task/project/member event)
          │
          ▼
  Look up recipient email address(es) from MongoDB
          │
          ▼
  Call sendEmail(to, subject, htmlTemplate)
  [async, non-blocking — does NOT await]
          │
          ▼
  Nodemailer creates SMTP transport with Gmail credentials
          │
          ▼
  Sends HTML email via Gmail SMTP
          │
          ◄───── Response already sent to client (not waiting for email)


─── ACTIVITY LOGGING FLOW ────────────────────────────────────────────────────

  Controller action succeeds (e.g. task created)
          │
          ▼
  logActivity(workspaceId, actorId, action, entityType, entityName, metadata)
  [async, non-blocking]
          │
          ▼
  ActivityModel.create({ ... }) → saved to MongoDB
          │
          ▼
  Activity Feed component polls every 10 seconds:
    GET /api/analytics/workspace/:wid/activity
          │
          ▼
  MongoDB find({ workspaceId }) → populate actor (name, profilePicture)
          │
          ▼
  Rendered as timeline with actor avatar, action badge, timestamp

10. AUTHENTICATION & SECURITY
================================================================================

  • All API routes (except /api/auth/*) are protected by isAuthenticated middleware
  • isAuthenticated checks for a valid session in the cookie
  • Sessions are managed by cookie-session with a secret key
  • Passwords are hashed using bcrypt (saltRounds: 10) before storage
  • Google OAuth 2.0 handled by Passport.js (passport-google-oauth20)
    — User is upserted on Google login (email as unique key)
  • All inputs validated by Zod schemas before processing
  • Invite codes are UUIDs, reset on demand
  • CORS is configured to only allow requests from FRONTEND_ORIGIN
  • SMTP credentials stored in environment variables, never in source code

SESSION LIFECYCLE
  • Login: passport.js creates session, stores userId
  • Session cookie sent with every subsequent request
  • isAuthenticated reads session → attaches user to req.user
  • Logout: session.destroy()

11. PERMISSION SYSTEM
================================================================================

Roles: OWNER > ADMIN > MEMBER

Permissions assigned per role:

  OWNER permissions:
    VIEW_ONLY, ADD_MEMBER, CHANGE_MEMBER_ROLE, REMOVE_MEMBER,
    CREATE_PROJECT, EDIT_PROJECT, DELETE_PROJECT,
    CREATE_TASK, EDIT_TASK, DELETE_TASK,
    EDIT_WORKSPACE, DELETE_WORKSPACE, CHANGE_WORKSPACE_SETTINGS

  ADMIN permissions:
    VIEW_ONLY, ADD_MEMBER, CHANGE_MEMBER_ROLE, REMOVE_MEMBER,
    CREATE_PROJECT, EDIT_PROJECT,
    CREATE_TASK, EDIT_TASK, DELETE_TASK

  MEMBER permissions:
    VIEW_ONLY, CREATE_TASK

SPECIAL RULE — ASSIGNEE BYPASS:
  In updateTaskController, before enforcing EDIT_TASK permission,
  the controller checks: isAssignee = (task.assignedTo._id === req.user._id)
  If the request only changes the "status" field and the user IS the assignee,
  the EDIT_TASK permission check is skipped.
  This allows members to mark their own tasks complete without admin rights.

13. AI INTEGRATION
================================================================================

PROVIDER: Google Generative AI (Gemini 1.5 Flash)
LIBRARY: @google/generative-ai npm package

HOW IT WORKS
  1. User fills the task title in Create Task form
  2. User clicks "✨ AI Enhance" button
  3. Frontend calls POST /api/ai/enhance-task with:
       { title, projectName, workspaceName, existingDescription }
  4. Backend builds a structured prompt and calls Gemini 1.5 Flash
  5. Gemini returns a JSON object (description, suggestedPriority,
     estimatedHours, checklistItems)
  6. Frontend auto-fills the description field and updates priority
  7. Suggested subtasks displayed in a violet card below the textarea

PROMPT DESIGN
  — Model is told it is assisting on Team Sync platform
  — Context includes: task title, project name, workspace name,
    and any existing description (for enhancement not replacement)
  — Instructed to return clean JSON (no markdown code fences)
  — Backend strips markdown if model wraps in code fence anyway
  — Falls back gracefully if JSON parse fails (uses raw text as description)

TASK SUGGESTION ENDPOINT
  POST /api/ai/suggest-tasks
  Input: partial task title, optional project name
  Output: array of 4 specific, contextual task title completions
  Use case: Autocomplete / inspiration for task naming

15. TEAM BURNOUT RADAR
================================================================================

PURPOSE
  Proactively detect and prevent team burnout by computing a health score
  for each member based on their current task load.

HEALTH SCORE FORMULA
  Start with base score of 100.

  DEDUCTIONS:
    overdue tasks          × 15   (highest weight — overdue = most stressful)
    high-priority open     × 8    (significant concern)
    load ratio > 1.5x avg  − 20   (significantly above everyone else)
    load ratio > 1.2x avg  − 10   (moderately above team average)

  Score is clamped to [0, 100].

STATUS BANDS
  80 – 100  →  Healthy    (green)   — Manageable load
  50 –  79  →  At Risk    (yellow)  — Monitor this person
   0 –  49  →  Overloaded (red)     — Send email alert

AUTO-ALERT LOGIC
  • If score < 50, send burnout alert email to the member
  • Rate-limited to once per 24 hours per member (in-memory timestamp map)
  • Alert email includes: score, open tasks, overdue count, high-priority count
  • Email encourages the member to speak to their admin

UI DESIGN
  • Card per member with:
    — Avatar + name + role
    — SVG circular progress ring (color-coded, animated)
    — Status badge (Healthy / At Risk / Overloaded)
    — Task stats: X open, Y overdue, Z high-priority
  • Team average score banner at top
  • Sorted by health score (lowest = most at risk — shown first)
  • Tooltip showing breakdown on hover
  • Refresh button + auto-refreshes every 30 seconds

16. REAL-TIME DATA SYNC
================================================================================

Team Sync uses intelligent polling via TanStack Query's refetchInterval
to keep data fresh without WebSockets:

  Data Type              | Refetch Interval
  ─────────────────────────────────────────
  Activity Feed          | 10 seconds
  Team Burnout Radar     | 30 seconds
  Tasks List             | 10 seconds
  Workspace Analytics    | 10 seconds
  Projects               | 30 seconds

  On mutation (create/update/delete), TanStack Query immediately invalidates
  the relevant cache keys, causing an instant re-fetch — so changes made by
  any user appear to all users within the next poll interval.

19. DEPLOYMENT NOTES
================================================================================

BACKEND DEPLOYMENT (e.g., Railway, Render, Heroku)
  1. Set all environment variables in the hosting platform dashboard
  2. Ensure MongoDB Atlas Network Access allows the server's IP
  3. Run: npm run build && npm start
  4. Google OAuth: add production callback URL to Google Cloud Console
  5. Update FRONTEND_ORIGIN to production frontend URL

FRONTEND DEPLOYMENT (e.g., Vercel, Netlify)
  1. Set VITE_API_BASE_URL to production backend URL
  2. Set VITE_GOOGLE_CLIENT_ID
  3. Run: npm run build
  4. Deploy /dist folder

IMPORTANT NOTES
  • Never commit .env files to version control
  • Rotate SESSION_SECRET and GOOGLE_CLIENT_SECRET periodically
  • Gmail App Passwords can be revoked and regenerated at any time
  • Gemini API free tier: 15 requests/minute, 1 million tokens/day
  • MongoDB Atlas free tier: 512 MB storage
